## 一、严重 Bug

### #1 WebScrapingTool — 截断提示永远显示 8000 字符

**问题描述**：网页抓取工具在内容超过 8000 字符时截断，但截断提示中的"原始长度"永远是 8000，因为 `substring` 先执行，`length()` 返回的是截断后的长度。

```java
// 错误代码
if (bodyText.length() > MAX_BODY_LENGTH) {
    bodyText = bodyText.substring(0, MAX_BODY_LENGTH)
            + "\n\n[... total length: " + bodyText.length() + " characters]";
    //                                        ^^^^^^^^^^^^^^^^ 永远是 8000！
}
```

**根因分析**：这是一个典型的**操作顺序错误**。`substring` 是破坏性操作（返回新字符串），原字符串引用被覆盖，后续对 `bodyText` 的任何调用都基于新字符串。开发者写这段代码时没有意识到 Java String 的不可变性——`substring` 不修改原字符串，但**赋值** `bodyText = ...` 替换了引用。

**解决方案**：在截断前保存原始长度。

```java
int originalLength = bodyText.length();
if (originalLength > MAX_BODY_LENGTH) {
    bodyText = bodyText.substring(0, MAX_BODY_LENGTH)
            + "\n\n[... content truncated, original length: "
            + originalLength + " characters]";
}
```

**为什么这样解决**：这是最直接的修复——不改变数据结构和方法签名，只在正确的时间点取值。`MAX_BODY_LENGTH` 是常量（8000），所以不需要额外变量；但原始长度是变量，必须在 mutation 前捕获。

**延伸思考**：这个 bug 能通过**单元测试**轻松发现。如果有一个测试用例传入 9000 字符的内容，断言输出中包含 "original length: 9000"，就永远不会上线。

### #2 KnowledgeDocumentLoader — 文件名解析提取到无意义的字符

**问题描述**：

```java
String status = filename.substring(filename.length() - 6, filename.length() - 4);
```

这段代码从文件名中硬抠 2 个字符作为文档的 `status` 元数据。不同文件名长度不同，提取到的总是乱码：
- `AI-Agent-基础概念与架构.md` → "与构"（无意义）
- `RAG-检索增强生成技术.md` → 乱码

更严重的是，这个 `status` 字段被 `RagCustomAdvisorFactory` 用于检索过滤（`Filter.Expression.eq("status", status)`），导致过滤功能完全不可用。

**根因分析**：这是一个**数据建模错误**。元数据应该由文档作者显式声明，而不是从文件名按字节位置硬抠。文件名是给人看的，元数据是给程序读的——不能把两个用途混在一起。更深层的问题是没有思考"这个字段的业务含义是什么"——它应该是文档分类标签（如 "agent-basics"、"rag"、"react-pattern"），而不是中文字符片段。

**解决方案**：在 Markdown 文档头部引入 YAML Front Matter 声明元数据：

```yaml
---
category: agent-basics
tags: [AI Agent, LLM, Memory, Tools, Planning, Spring AI]
summary: AI Agent 基础概念、核心组件与架构模式介绍
---
```

并在 `KnowledgeDocumentLoader` 中新增 FrontMatter 解析器：

```java
private static class FrontMatter {
    String category;
    List<String> tags;
    String summary;

    static FrontMatter parse(String content) {
        // 解析 --- 分隔的 YAML 头部
        if (!content.startsWith("---")) return null;
        int end = content.indexOf("---", 3);
        // ... 逐行解析 key: value
    }
}
```

同时新增 `GET /api/ai/rag/documents` 端点，让前端能展示知识库文档目录。

**为什么这样解决**：YAML Front Matter 是静默文档站（Jekyll、Hugo、Hexo）的通用标准，具有自描述性、可扩展性、人机可读的优点。相比文件名 hack，它：
- 不依赖文件名格式（文件名可以自由命名）
- 支持未来扩展更多字段（如 author、date、version）
- 能被任何解析器读取（不绑定 Java 实现）

**延伸思考**：元数据的设计原则——**显式优于隐式**。从文件名、路径、时间戳推断出的元数据永远不可靠，让作者显式声明才是正道。

### #3 RAG 知识库问答忽略用户 Provider 配置

**问题描述**：`KnowledgeBaseService` 的构造函数直接注入 `dashscopeChatModel`，所有 RAG 对话都走 DashScope。但用户在 Settings 页面切换到 OpenAI 后，Agent 用 OpenAI，RAG 仍然用 DashScope。如果用户没有 DashScope Key，Agent 正常但 RAG 直接挂掉。

**根因分析**：这是一个**架构设计问题**——RAG 模块和 Agent 模块对 LLM Provider 的获取方式不一致。

```
Agent:  DynamicChatModelFactory → 读取 session 对应的 ModelConfig → 动态创建
RAG:   直接注入 dashscopeChatModel（构造时写死）→ 永远用 DashScope
```

Spring 的依赖注入让 `dashscopeChatModel` 看起来很"方便"——构造函数参数自动获得，不需要手动管理。但这种便利性引出了一个隐式假设："RAG 总是用 DashScope"。一旦需求变化（支持多 Provider），这个假设就碎了。

**解决方案**：为 RAG 端点添加 `sessionId` 参数，通过 `DynamicChatModelFactory.createChatModel(config)` 动态创建 ChatModel：

```java
// AiController.java
private ChatModel resolveRagChatModel(String sessionId) {
    if (sessionId == null || sessionId.isBlank()) return null;
    AgentSession session = sessionManager.getSession(sessionId);
    if (session == null) return null;
    ModelConfig config = session.getModelConfig();
    if (config == null || !config.isValid()) return null;
    return chatModelFactory.createChatModel(config); // 只返回 ChatModel，不含 ChatClient
}

// KnowledgeBaseService.java — 新增重载方法
public String doChatWithRag(String message, String chatId, ChatModel customModel) {
    ChatClient client = (customModel != null)
        ? resolveChatClient(chatId, customModel)  // 用用户的 Provider
        : defaultChatClient;                       // 回退默认
    // ...
}
```

**为什么这样解决**：不改动 `KnowledgeBaseService` 的核心逻辑，只增加一个可选的重载路径。`customModel != null` 时走用户配置，`null` 时回退默认。这个设计遵循**开闭原则**——对扩展开放（新增 `sessionId` 参数路径），对修改关闭（原有代码无需改动）。

**延伸思考**：依赖注入的便利性常常掩盖了"默认值就是写死的"这一事实。凡是注入 `Foo defaultFoo` 的地方，都应该问一句——"这个默认值什么时候需要被替换？替换路径存在吗？"

## 二、安全与架构

### #5 CORS 配置过于宽松

**问题描述**：`allowedOriginPatterns("*")` + `allowCredentials(true)` 组合允许任何网站携带 Cookie 跨域请求 API。

**根因分析**：开发阶段为了方便（前端可能在任意端口），使用了最宽松的 CORS 配置，但没有区分环境。CORS 是浏览器的同源策略的"例外放行单"，配置越宽松，安全风险越大。

**解决方案**：通过配置文件注入允许的域名：

```java
@Value("${app.cors.allowed-origins:http://localhost:3000}")
private String allowedOrigins;

// 支持逗号分隔的多域名
config.addAllowedOriginPattern(allowedOrigins.split(","));
```

**为什么这样解决**：环境差异化配置是 Spring Boot 的标准实践。`application.yml` 中开发环境设为 `http://localhost:3000`，生产环境通过环境变量覆盖为实际域名。单一默认值确保开发体验不受影响，可配置性确保生产安全。

### #6 LoggingAdvisor 的 NPE 风险

**问题描述**：

```java
chatClientResponse.chatResponse().getResult().getOutput().getText()
```

链式调用中任何一个中间方法返回 `null` 都会导致 NPE。在 Function Calling 场景下，LLM 只返回 toolCalls 不返回 text（`getText()` 返回空），但如果某个 Provider 实现中 `getResult()` 返回 null，就会崩。

**根因分析**：链式调用假设每一步都成功返回非 null 值。这个假设在"主线场景"（LLM 正常返回文本）下成立，但在边缘场景（只返回 toolCalls、Provider 异常、空响应）下不成立。防御性编程的核心原则是：**永远不要信任外部系统的返回值**。

**解决方案**：逐级空检查：

```java
private void observeAfter(ChatClientResponse chatClientResponse) {
    var response = chatClientResponse.chatResponse();
    if (response != null && response.getResult() != null) {
        var output = response.getResult().getOutput();
        if (output != null) {
            log.info("AI Response: {}", output.getText());
        }
    }
}
```

**为什么这样解决**：Advisor 是拦截器链中的一环，不应该因为日志记录失败而中断 Agent 执行。最坏情况（所有值都是 null）也只是静默跳过，不影响业务逻辑。

### #7 SSE 连接无心跳导致长时间执行超时

**问题描述**：`SseEmitter` 超时设为 5 分钟，但如果工具执行超过 5 分钟（如大文件下载），连接会断开，前端显示连接错误，但 Agent 仍在后端运行。

**根因分析**：SSE 是 HTTP 长连接。很多代理服务器（Nginx、负载均衡器）会静默关闭长时间无数据传输的连接。没有心跳意味着即使 Agent 在正常工作，连接也可能被中间网络设备切断。

**解决方案**：在 `BaseAgent.runStream()` 和 `runStreamStructured()` 中增加心跳机制：

```java
ScheduledExecutorService heartbeatExecutor = Executors.newSingleThreadScheduledExecutor();
ScheduledFuture<?> heartbeat = heartbeatExecutor.scheduleAtFixedRate(() -> {
    try {
        sseEmitter.send(SseEmitter.event().comment("ping"));
    } catch (IOException ignored) { /* connection already closed */ }
}, 60, 60, TimeUnit.SECONDS);

// 在 finally / onTimeout / onCompletion 中清理
finally {
    stopHeartbeat(heartbeat, heartbeatExecutor);
    this.cleanup();
}
```

**为什么这样解决**：SSE 协议规定以 `:` 开头的行是注释，客户端会忽略。用 `SseEmitter.event().comment("ping")` 发送空事件，既满足心跳需求，又不影响前端解析。`ScheduledExecutorService` 确保每 60 秒固定发送，资源在连接关闭时正确释放。

### #8 全局异常处理缺失

**问题描述**：所有 Controller 方法各自处理异常，前端收到的错误格式不一致：有时是纯文本、有时是 JSON、有时是 SSE error 事件。

**根因分析**：异常处理是横切关注点（cross-cutting concern），每个 Controller 方法不应该各自实现。没有统一异常处理导致两个问题：前端需要处理多种错误格式（增加复杂度），后端错误信息泄露给前端（安全风险）。

**解决方案**：创建 `GlobalExceptionHandler.java`（`@RestControllerAdvice`）：

```java
@ExceptionHandler(IllegalArgumentException.class)
public ResponseEntity<Map<String, Object>> handleBadRequest(IllegalArgumentException e) {
    return ResponseEntity.badRequest().body(Map.of("error", e.getMessage(), "status", 400));
}

@ExceptionHandler(Exception.class)
public ResponseEntity<Map<String, Object>> handleGeneral(Exception e) {
    log.error("Unhandled exception", e); // 服务端记录完整堆栈
    return ResponseEntity.status(500).body(Map.of("error", "Internal server error", "status", 500));
    //                                      ^^^^^^^^^^^^^^^^ 前端只看到通用消息，不泄露细节
}
```

**为什么这样解决**：`@RestControllerAdvice` 是 Spring MVC 的标准化全局异常处理方案。统一的 JSON 错误格式（`{"error": "...", "status": N}`）让前端可以写一个通用的错误处理逻辑。生产环境中，详细错误信息写日志、通用消息返前端——兼顾调试和安全性。

### #9 BaseAgent 中的 instanceof 违反依赖倒置原则

**问题描述**：`BaseAgent.emitFinalAnswerIfAvailable()` 使用 `this instanceof ToolCallAgent` 来判断行为。父类代码依赖了具体的子类类型。

**根因分析**：`instanceof` 在父类中判断子类类型，破坏了面向对象的**依赖倒置原则**——高层模块（BaseAgent）不应该依赖低层模块（ToolCallAgent）。应该依赖抽象（模板方法）。这会导致：新增一个不继承 ToolCallAgent 的 Agent 子类时，`emitFinalAnswerIfAvailable()` 静默跳过，行为不可预期。

**解决方案**：使用模板方法模式：

```java
// BaseAgent.java
protected String getFinalAnswerText() {
    return null; // 默认：无最终答案
}

// ToolCallAgent.java
@Override
protected String getFinalAnswerText() {
    return lastThinkingText;
}
```

**为什么这样解决**：模板方法模式（Template Method）是开闭原则的经典实践。BaseAgent 定义算法的骨架（`emitFinalAnswerIfAvailable()`），子类通过重写 `getFinalAnswerText()` 提供具体实现。新增任何子类只需重写这个方法，不修改 BaseAgent。

### #10 RAG 会话记忆未持久化（与 Agent 不一致）

**问题描述**：Agent 会话用 `FileBasedChatMemory`（Kryo 文件持久化），RAG 会话用 `InMemoryChatMemoryRepository`（纯内存）。用户在不同页面之间的"对话连续性"体验不一致。

**根因分析**：两个模块独立开发、独立选择记忆方案，缺乏架构层面的统一设计。Agent 模块需要跨请求保持对话上下文（因为 ReAct 循环多次与 LLM 交互），所以很早就做了持久化。RAG 模块当时没考虑"用户可能关掉页面明天再来问"的场景。

**解决方案**：统一使用 `FileBasedChatMemory`：

```java
// KnowledgeBaseService 构造函数
FileBasedChatMemory persistentMemory = new FileBasedChatMemory(
    System.getProperty("user.dir") + "/tmp/rag-memory");
MessageWindowChatMemory chatMemory = MessageWindowChatMemory.builder()
    .chatMemoryRepository(persistentMemory)
    .maxMessages(20)
    .build();
```

**为什么这样解决**：`FileBasedChatMemory` 同时实现了 `ChatMemory` 和 `ChatMemoryRepository` 接口，Agent 和 RAG 都对接同一套持久化方案。Kryo 序列化虽不是生产级方案（生产应用应用 Redis/数据库），但满足了当前"本地开发 + 重启可恢复"的需求，且不引入额外依赖。

### #11 TerminalOperationTool 白名单包含高危命令

### #12 3 处 e.printStackTrace() 应改为日志框架

**问题描述**：`ReActAgent` 和 `FileBasedChatMemory` 中使用 `e.printStackTrace()` 输出到 stderr。

**根因分析**：`printStackTrace()` 不经过日志框架，输出到 stderr，无法按级别过滤、无法写入文件、无法聚合。在生产环境中，stderr 是多线程共享的，不同线程的输出会交错，难以阅读。

**解决方案**：全部替换为 `log.error("描述信息", e)`。

**为什么这样解决**：SLF4J + Logback 是 Spring Boot 的标准日志方案，支持分级、文件轮转、异步写入。统一的日志格式让问题排查更高效。

### #13 从磁盘恢复会话时丢失 Provider 配置

**问题描述**：`getOrCreateSession()` 第 194 行用 `ModelConfig.defaultConfig()` 恢复磁盘会话，丢失了用户之前配置的 Provider 和 API Key。


| 方案 | 优点 | 缺点 |
|------|------|------|
| 扩展 .meta.json（推荐） | 复用现有机制，改动小 | API Key 需重新输入 |

### #19 Agent 完成任务后重复回答不调用 terminate

**问题描述**：Agent 使用 `writeFile` 创建文件成功后，不立即调用 `doTerminate` 终止，而是连续 4 步重复"任务已完成，文件已创建"，直到第 6 步才终止。

**根因分析**：问题有三个层面：

```
1. 提示词层面：NEXT_STEP_PROMPT 只在第一步注入一次
   → LLM 完成工具调用后"忘记"了 terminate 这个选项
   → 类比：厨师做完菜忘了说"出菜"，一直在重复"菜做好了"

2. 代码层面：think() 返回 false（无工具调用）时
   没有设置 FINISHED 状态，ReAct 循环在空转
   → 代码说"你没调工具，继续想"，LLM 说"我想好了，不需要调工具"
   → 僵局！

3. 防御层面：没有检测"重复回答"的机制
   → 一模一样的文本说了 4 次，系统毫无察觉
```

**解决方案**（三层防护）：

**第 1 层 — 提示词加固**（DevAssistantAgent.java）：
```
CRITICAL: 任务一旦完成，必须立即调用 doTerminate 终止！
不要重复描述已完成的工作，不要问用户确认。
```

**第 2 层 — 连续空转检测**（ToolCallAgent.java）：
```java
// 连续 N 步 think() 返回"无工具调用" → 自动终止
private int consecutiveNoToolCallSteps = 0;
private static final int MAX_NO_TOOL_CALL_STEPS = 2;

if (toolCallList.isEmpty()) {
    consecutiveNoToolCallSteps++;
    if (consecutiveNoToolCallSteps >= MAX_NO_TOOL_CALL_STEPS) {
        setState(AgentState.FINISHED);  // 不等 LLM 了，直接结束
    }
} else {
    consecutiveNoToolCallSteps = 0;  // 有工具调用，重置计数
}
```

**第 3 层 — 重复内容检测**（ToolCallAgent.java）：
```java
// 当前步的思考文本 == 上一步 → 陷入复读 → 自动终止
if (currentThinking.equals(previousThinkingText)) {
    setState(AgentState.FINISHED);
}
this.previousThinkingText = currentThinking;
```

**实测效果**：
| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 文件创建 | 6 步（4 步重复） | **2 步**（writeFile → terminate） |
| Web 搜索 | 之前无限循环到 maxSteps | **12 步**正常终止 |

**为什么这样解决**：三层防护对应三种失效模式：
- 提示词层：降低 LLM "忘记"的概率（预防）
- 空转检测层：当 LLM 不调工具时强制结束（兜底）
- 重复检测层：当 LLM 陷入复读循环时打断（兜底）

关键是**不依赖 LLM 的自觉性**——LLM 的输出是不可靠的，代码必须有独立的终止判断逻辑。

### #20 历史会话无限堆积

**问题描述**：持久化目录 `tmp/agent-memory/` 中积累了 28 个会话文件（含大量测试遗留："你是谁？" × N），且只有内存 30 分钟超时，磁盘文件永不过期。

**根因分析**：会话管理有两层生命周期：

```
内存层：30 分钟超时 → Session 从 ConcurrentHashMap 移除 ✅
磁盘层：永不过期（注释写死了 "disk files preserved"）❌
```

设计意图是"用户可能关闭页面，过几天回来继续对话"，但实际效果是测试垃圾 + 一次性会话堆积成山。这是**没有在设计与实际使用场景之间做出权衡**的典型表现。

**解决方案**（双重保障）：

```java
// 保障 1：磁盘过期（7 天未修改的文件自动删除）
private static final long DISK_TTL_MS = 7 * 24 * 60 * 60 * 1000;

// 保障 2：数量上限（最多保留 20 个会话，超出淘汰最旧的）
private static final int MAX_DISK_SESSIONS = 20;
```

在定时清理任务中同时检查内存和磁盘，在创建新会话时触发配额检查：

```
内存清理（每 5 分钟）：30 分钟无活动 → 持久化后移除
磁盘清理（每 5 分钟）：7 天未修改 → 删除 .kryo + .meta.json
配额检查（每次创建）：超过 20 个 → 按 lastModified 淘汰最旧的
```

**为什么这样解决**：时间过期处理"自然老化"，数量上限处理"突发堆积"。两者互补：
- 正常情况下，7 天 TTL 足够了（大部分人一周内会回来）
- 异常情况下（一天创建 50 个测试会话），20 上限兜底
- LRU 淘汰策略（按最后修改时间）保护活跃会话，清理冷数据

参数设计（7 天 / 20 个）是根据学习项目的使用频率估算的——平衡了"保留有用会话"和"防止磁盘浪费"。

### #22 自动终止步骤导致回复内容重复显示（P0 🔴）

这是用户直接感知到的重复 bug。完整执行链路：

```
Step 1:  think() → LLM 返回 "您之前提到自己是小羊..."（有文本）→ 无工具调用
         step() → getLastAssistantText() → "您之前提到自己是小羊..." ✅
         step_end → finalResponse += 此文本

Step 2:  think() → LLM 再次被调用，返回空文本 + 无工具调用
                 → consecutiveNoToolCallSteps=2 → state=FINISHED
         step() → getLastAssistantText() → 倒序跳过空消息 → 找到 Step1 的文本 ❌
         step_end → finalResponse += 相同文本（重复！）

final_answer: lastThinkingText 为空（Step2 返回空文本）
agent_finish: reply = finalAnswerText || finalResponse.trim()
            = "" || "您之前提到...\n您之前提到..."  ← 同一内容出现两次！
```

**根因涉及 3 个文件**：

| 文件 | 行号 | 问题 |
|------|------|------|
| `ToolCallAgent.java` | 139 | 无工具调用时将空 `AssistantMessage` 加入 messageList |
| `ReActAgent.java` | 64-76 | `getLastAssistantText()` 倒序查找，跳过空消息后返回上一步的旧文本 |
| `AgentChat.vue` | 190-194 | `finalAnswerText` 为空时回退到 `finalResponse`，其中拼了两份相同文本 |

**解决方案**（推荐，改 2 处）：

1. `BaseAgent.java` 循环中：step 后若 state 已变为 FINISHED 则立即 break，不发射重复 step_end
```java
String stepResult = step();
sendJsonEvent(sseEmitter, AgentEvent.stepEnd(stepNumber, stepResult));
if (getState() == AgentState.FINISHED) {
    break;  // ← 自动终止不是"产生新回复"，不发射多余的 step_end
}
```

2. `ToolCallAgent.think()` 中：自动终止时不再将空 `AssistantMessage` 加入 messageList

**备选方案**：

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A（推荐）| 后端 break + 不存空消息 | 根因修复 | 需改 2 处 |
| B | 仅前端去重 `finalResponse` | 改动最小 | 治标，前端逻辑越来越 hack |
| C | 改为单步即终止（移除 `MAX_NO_TOOL_CALL_STEPS`） | 简化逻辑 | 可能误终止需要多步思考但无工具调用的场景 |
| D | `getLastAssistantText()` 检查是否与前一次相同 | 1 行改动 | 治标，其他类似场景仍可能触发 |

**🔧 2026-07-03 补丁 — 初版修复不完整**：

初版修复（break + 不存空消息）上线后发现两个遗留问题：

1. **重复文本仍然出现**：`break` 阻止了 Step 3+，但 Step 2 的 `step_end` 仍然携带过期文本——`ReActAgent.step()` 在 `think()` 返回 false 后调用 `getLastAssistantText()`，倒序跳过当前步的空消息后找到 Step 1 的文本，同一内容被发射两次 `step_end`。

2. **前端思考链 Step 2 一直转圈**：因为跳过了 `step_end` 发射（补丁的初版尝试），前端 `ThinkingChain` 收到 `step_start(2)` 后永远等不到对应的 `step_end`，spinner 持续旋转。

**补丁修复**（2 个文件）：

`ReActAgent.java:44-49` — 自动终止时不返回过期文本：
```java
if (!shouldAct) {
    // 自动终止时 think() 已将状态设为 FINISHED，
    // getLastAssistantText() 会跳过当前步的空消息找到上一步的文本
    if (getState() == AgentState.FINISHED) {
        return null;  // 让 BaseAgent 发射空内容的 step_end
    }
    // ...原有逻辑
}
```

`BaseAgent.java` (3 处 `run()`/`runStream()`/`runStreamStructured()`) — 始终发射 step_end，空内容兜底：
```java
// 自动终止时 stepResult 为 null，发射空内容的 step_end：
// - 前端 finalResponse 不累积空内容（evt.content 为 falsy）
// - 前端 ThinkingChain 收到 step_end 关闭当前步骤（停止转圈）
sendJsonEvent(sseEmitter, AgentEvent.stepEnd(stepNumber,
        stepResult != null ? stepResult : ""));
```

**设计要点**：`step_end` 有两重语义——① 携带文本（供 `finalResponse` 拼接）② 关闭步骤（供 `ThinkingChain` 状态管理）。初版修复只考虑了①，忽略了②。补丁让两者解耦：空内容满足②，不影响①。

**🔧 2026-07-03 第二次补丁 — 方案 A：聊天面板回复碎片化**：

补丁一上线后，用户发现新问题：**思考链显示的内容合理，但聊天面板的回复不合理**。

**现象**（以第 4 轮对话"我是谁"为例）：

```
聊天面板显示:  "这样我可以更好地为你服务。"     ← 不完整的续句！
思考链显示:    Step 1 "你之前提到你是小航..."    ← 完整的句子
              Step 2 "这样我可以更好地为你服务。" ← 续句
```

**根因链路**（3 层）：

| 层 | 问题 | 详细 |
|----|------|------|
| ① 触发层 | LLM 被不必要地调用了第二次 | Step 1 已返回完整文本 + 无工具，但 `MAX_NO_TOOL_CALL_STEPS=2` 不区分"有文本"和"空文本"，统一给了第二次机会。LLM 看到自己上一步的回复，继续"补全"输出续句 |
| ② 数据层 | 自动终止时非空文本被丢弃 | Step 2 续句文本不为空，但因自动终止分支跳过了 `messageList.add()`，文本既不在对话历史中，也不被 `getLastAssistantText()` 找到 |
| ③ 展示层 | `final_answer` 只取最后一步 | `getFinalAnswerText()` 返回 `lastThinkingText`（Step 2 的续句），而非完整的 LLM 回复 |

**这与主流产品的对比**：

| 产品 | 终止判断 | 会出现碎片吗？ |
|------|---------|:--:|
| **Claude** | `stop_reason == "end_turn"` | ❌ API 语义明确——文本 = 结束，从不二次调用 |
| **ChatGPT** | `finish_reason == "stop"` | ❌ 同上 |
| **GLM** | `finish_reason == "stop"` | ❌ 同上，`AgentGLM` 开源项目甚至更激进——不调工具直接终止 |
| **我们的 Agent（修复前）** | `consecutiveNoToolCallSteps >= 2` | ✅ 框架不信任 LLM 的 stop 信号，擅自再问一次 |

**修复（方案 A — 对齐主流设计）**：

核心思路：**信任 LLM 的意图信号**——"有文本 + 无工具 = 最终回复"，和 Claude/ChatGPT/GLM 的行为一致。

`ToolCallAgent.think()` — 区分"有文本"和"空文本"两种情况：
```java
if (toolCallList.isEmpty()) {
    consecutiveNoToolCallSteps++;
    if (StrUtil.isNotBlank(thinkingText)) {
        // 有文本 → 这就是最终答案，立即终止并保存消息（对齐 Claude/ChatGPT/GLM）
        getMessageList().add(assistantMessage);
        setState(AgentState.FINISHED);
    } else if (consecutiveNoToolCallSteps >= MAX_NO_TOOL_CALL_STEPS) {
        // 无文本且连续空转达到阈值 → 强制终止（仅兜底异常场景）
        setState(AgentState.FINISHED);
    }
    // 无文本且未达阈值：不给 LLM 回复，继续下一步（给一次重试机会）
}
```

`ToolCallAgent.getFinalAnswerText()` — 优先从 messageList 取，兜底 lastThinkingText：
```java
@Override
protected String getFinalAnswerText() {
    // 优先从 messageList 获取（有文本时总是先存到 messageList 再设 FINISHED）
    List<Message> messages = getMessageList();
    for (int i = messages.size() - 1; i >= 0; i--) {
        if (messages.get(i) instanceof AssistantMessage assistantMsg) {
            String text = assistantMsg.getText();
            if (StrUtil.isNotBlank(text)) return text;
        }
    }
    return lastThinkingText; // 兜底（如 reasoningContent）
}
```

**修改后的循环逻辑**：

```
think() 返回 → 有工具调用？
  ├─ YES → act() 执行工具 → 继续循环
  └─ NO  → 有文本？→ FINISHED（这就是最终回复，对齐 Claude/ChatGPT/GLM）
           └─ 无文本？→ 计数器++ → 满2次？→ 强制 FINISHED（兜底异常）
```

**方案对比（评估过的 6 种方案）**：

| 方案 | 描述 | 根因修复 | 改动量 | 新问题风险 |
|------|------|:--:|:--:|:--:|
| **A（选用）** | 有文本立即终止，无文本兜底 | ★★★ | ★★★ | 低（LLM 第一步输出过渡性文本的概率低） |
| B | 仅改 `getFinalAnswerText()` 数据源 | ★ | ★★★ | 高（聊天仍显示碎片） |
| C | 累积拼接所有 thinkingText | ★★ | ★★ | 中（可能拼接无关内容） |
| D | 前端优先用 `finalResponse` | ★ | ★★ | 高（包含工具原始输出，格式丑） |
| E | 累加确认状态机 | ★★ | ★ | 高（复杂度显著增加） |

### #23 空 AssistantMessage 和 nextStepPrompt 污染对话历史（P2 🟡）

**问题**：messageList 中混入了两种不该出现的内容：

**污染 1 — 空 AssistantMessage**：
`ToolCallAgent.think()` 第 139 行，无工具调用时将 `assistantMessage`（文本通常为空）加入 messageList。这些空消息被持久化到 `.kryo` 文件，并在多轮对话时全量回放给 LLM，浪费上下文窗口。本次测试会话 12 条消息中有 3 条是空消息。

**污染 2 — nextStepPrompt 伪装成 UserMessage**：
`ToolCallAgent.think()` 第 95 行：
```java
UserMessage userMessage = new UserMessage(getNextStepPrompt());
getMessageList().add(userMessage);
```

系统指令以"用户消息"的身份混入对话历史。前端被迫用字符串匹配来过滤（`AgentChat.vue:143`），但后端仍将其发送给 LLM。

```json
{"type": "USER", "text": "Based on the current task progress, determine the next action..."}
```

**解决方案**：
- **空消息**：在 `think()` 中，若无工具调用且文本为空，不加入 messageList
- **nextStepPrompt**：改为追加到 system prompt 末尾（在 `Prompt` 构造时），而非作为消息历史

**为什么这样解决**：LLM 的 system prompt 是与 user message 在不同层级注入的——system prompt 不参与对话历史、不被持久化、不消耗消息配额。将 nextStepPrompt 从 messageList 移到 system prompt，既保持 LLM 收到相同的指令，又清理了对话历史。

### #24 Agent 重建后 getLastAssistantText() 可能返回上一轮的旧文本（P2 🟡）

**问题**：`ReActAgent.getLastAssistantText()`（第 64 行）的设计假设是"消息列表中最近的非空 AssistantMessage 就是当前步的思考结果"。但以下场景会导致误判：

1. Agent FINISHED → 重建 → loadSessionHistory → messageList 恢复
2. 新用户消息加入 → think() 返回空消息 → 空消息加入列表
3. getLastAssistantText() 跳过空消息 → 返回到**上一次对话**的 Assistant 回复

实际影响：虽然当前有 `consecutiveNoToolCallSteps` 兜底（2 步后强制终止），但第 2 步的 `step_end` 可能携带错误的文本。

**解决方案**：`think()` 返回 false 时，若 state 被设为 FINISHED，不再调用 `getLastAssistantText()`——直接用 `lastThinkingText` 作为 step 结果。或者给 `getLastAssistantText()` 增加一个"只查找当前 step 范围内的消息"的约束。

**状态**：与 #22 关联修复。#22 的 break 方案已能防止该问题产生用户可见影响。

### #25 messageList 无限增长，无上下文窗口管理（P2 🟡）

**文件**：`BaseAgent.java:65`

**问题**：`messageList` 只增不减。工具结果有 2000 字符截断（`MAX_TOOL_RESULT_LENGTH`），但**消息数量无上限**。每轮对话平均产生 4-6 条消息（user + assistant + tool call + tool response），20 轮后接近 100 条，远超多数 LLM 的上下文窗口。

实际观察：测试会话 3 轮对话产生 12 条消息，按此速度 15 轮就会达到 ~60 条。

**解决方案**：在 `think()` 构造 Prompt 前对 messageList 做滑动窗口裁剪：
```java
private static final int MAX_MESSAGE_COUNT = 50; // 保留最近 50 条

if (messageList.size() > MAX_MESSAGE_COUNT) {
    int keep = MAX_MESSAGE_COUNT;
    List<Message> trimmed = new ArrayList<>();
    trimmed.add(messageList.get(0));  // 保留第一条（通常是关键上下文）
    trimmed.addAll(messageList.subList(messageList.size() - keep + 1, messageList.size()));
    // 使用 trimmed 构造 Prompt
}
```

**为什么这样解决**：滑动窗口是最简单的上下文管理策略。保留第一条消息是为了保留"会话起点"（如用户最初的问题设定），保留最近 N 条是为了保持对话连续性。更高级的方案（如 LLM 摘要旧消息）可以作为后续优化。

**优先级说明**：当前测试轮次少（3 轮）不会触发，但作为学习项目理解"AI Agent 如何管理有限上下文窗口"的核心问题，值得提前设计。

### 核心教训

1. **编译是第一个测试**：任何涉及控制流（if/for/try）的修改后立即编译。`#18` 是一个 6 行的复制残留，如果修改后花 5 秒编译，就不会浪费几小时。

2. **LLM 的输出是不可靠的，代码必须有独立的判断逻辑**：`#19` 和 `#22` 的核心教训。LLM 可能"忘记"调 terminate、可能陷入复读、可能误解工具含义——甚至第 2 步返回空文本导致代码回退到上一步的旧文本。代码不能被动等待 LLM 做正确的事，必须主动检测异常并兜底。

3. **元数据应该显式声明，不要从命名规则推断**：`#2` 的核心教训。文件名、时间戳、路径都是脆弱的元数据来源——只要约定一变，代码就崩。`#13` 也是同样的问题：Config 应该在创建时显式持久化，而不是期望它永远留在内存中。

4. **便利的默认值往往掩盖了"写死"的真相**：`#3`、`#5`、`#11`、`#13` 都印证了这一点。每次使用默认值（Default Bean、Default Config、Default Allowed List）时，都要问"什么时候需要替换？替换路径畅通吗？"

5. **安全防线要层层设防**：`#5`（CORS）、`#11`（命令白名单）、`#7`（心跳）、`#19`（自动终止）都是从不同维度设立防线。单一防线总会失效，多层防线即使每层不完美，组合起来也能大幅降低风险。

6. **测试要真实**：静态审查发现了 17 个问题，但 `#19`（Agent 重复回答）和 `#20`（会话堆积）只有用真实 API Key 跑了完整流程才发现。`#21-#26` 更进一步：只有实际跟踪用户的多轮对话才能暴露 sessionId 生命周期、step_end 重复、消息污染等跨越前后端边界的问题。

7. **全链路追踪是诊断分布式状态 bug 的唯一手段**：`#22` 需要同时追踪 LLM 返回 → think/act 循环 → messageList 变更 → SSE 事件发射 → 前端 fallback 逻辑，跨越 5 个组件。任何一个环节单独看都是"正确"的——只有把全链路串起来，才能看到 Step 2 的 `getLastAssistantText()` 返回了 Step 1 的文本。

8. **工具描述不是注释，是 LLM 的使用手册**：`#31` 的核心教训。一句话的工具描述 = 给厨师的菜谱只写了"用锅炒"。Claude Code / ChatGPT 的工具描述长达数百词，包含使用时机、停止条件、反面示例、查询技巧。每新增一个工具，问自己：LLM 看完这个描述后，能不能回答"什么时候用、什么时候不用、什么时候停"？qwen-max 在 Function Calling 模式下倾向于"能搜就搜"，我们的工具描述必须比 Claude/GPT 的工具更详细，才能达到同样的行为控制效果。

## 六、第五轮：功能增强 — 对齐主流 Agent（2026-07-04）

前四轮解决了 29 个问题（正确性、安全性、健壮性、运行时行为），本轮聚焦于"我们的 Agent 和 Claude/ChatGPT 之间差了什么"。

### 对比方法论

以 Claude Code、ChatGPT、LangGraph、AutoGen 四个主流 Agent 为基准，从 20 个维度逐一对比，按"用户可感知程度 × 实现可行性"排序，选出 5 个最高优先级改进。

### #32 Esc 键停止 Agent（对标 Claude Code / ChatGPT）

**问题**：Agent 前端关闭 EventSource 后，后端 `CompletableFuture` 仍在线程池中运行直到自然结束。用户无法中断"疯狂搜索"。

**根因**：三个缺失——(1) 无取消 API 端点，(2) `CompletableFuture.runAsync()` 返回的 Future 未被存储无法 cancel，(3) 步骤循环不检查取消标志。

**解决方案**（3 文件）：
- `AgentSessionManager` 新增 `ConcurrentHashMap<String, AtomicBoolean> cancelFlags`，暴露 `requestCancel()` / `isCancelled()` 方法
- `AiController` 新增 `POST /ai/agent/cancel` 端点
- `BaseAgent` 新增 `volatile boolean cancelled` 字段，`run()` / `runStream()` / `runStreamStructured()` 三步循环均检查
- `ToolCallAgent.think()` 入口检查（LLM 调用可能阻塞 5-30s）
- 前端 `AgentChat.vue` 监听全局 `keydown` → `Esc` 键触发停止

**为什么选择 Esc 键而非 Stop 按钮**：Claude Code 和 ChatGPT 都用 Esc 中断，键盘操作更快，UI 更干净。

### #33 并行工具调用（对标 Claude/GPT Parallel Function Calling）

**问题**：LLM 在单次 `think()` 中返回了多个 `tool_call` 时（如 3 个不同搜索词），Agent 顺序执行，耗时为 sum(各工具)。这也是 #31 搜索死循环的助推因素——每个搜索占用一个独立 step，增加了 LLM "追加搜索"的机会。

**根因**：`ToolCallAgent.act()` 使用 `toolCallingManager.executeToolCalls()` 一次性执行所有工具，Spring AI 内部可能顺序执行。

**解决方案**（1 文件）：
- `act()` 重构：单工具保持不变，多工具时用 `CompletableFuture.supplyAsync()` 并行执行
- 每个工具独立匹配 `ToolCallback`，直接调用 `callback.call(toolInput)`
- 结果通过 `CompletableFuture.get(30s)` 收集，超时兜底
- `sameToolCallCount` 从 `HashMap` 改为 `ConcurrentHashMap` 确保线程安全

**预期效果**：3 个独立搜索从 3 步（3 次 LLM 调用）压缩到 1 步（1 次 LLM 调用 + 并行执行），搜索次数减少的同时执行速度提升 2-3x。

### #34 Token 级流式输出（对标 Claude/ChatGPT 打字机效果）

**问题**：`ToolCallAgent.think()` 使用 `ChatClient.call()` 同步阻塞，LLM 完整返回后一次性推送 `thinking` 事件。用户看到的是 5-30 秒的空白等待（只有一个"正在调用 LLM..."占位），体验极差。

**根因**：Agent 循环的 `step()` 是同步方法——`String stepResult = step()` 必须等完整返回。Spring AI 的 `stream()` API 未被使用。

**解决方案**（4 文件）：
- `AgentEvent` record 新增 `delta` 字段 + `thinkingDelta()` 工厂方法
- `ToolCallAgent.think()`：`.call()` 改为 `.stream().chatResponse()`，`doOnNext` 逐 chunk 发射 `thinking_delta`，`.blockLast()` 等待完整响应
- `ThinkingChain.vue`：新增 `thinking_delta` case，逐片追加 `step.thinking`
- `AgentChat.vue`：新增流式聊天消息——token 到达时实时创建/更新 `ai-streaming` 消息，`agent_finish` 时转为 `ai-final`

**关键设计**：内部流式推送 + 外部 `blockLast()` 同步等待。改动仅限 `think()` 内部，不改变 Agent 循环的同步本质。

**前置条件确认**：项目已有 `Flux` 支持（RAG 路径已用），`LoggingAdvisor` 和 `ReReadingAdvisor` 已实现 `StreamAdvisor` 接口。

### #35 增量摘要记忆（对标 LangChain ConversationSummaryMemory）

**问题**：`messageList` 超过 50 条后，代码执行滑动窗口裁剪——只保留首条 + 最近 49 条，中间消息直接丢弃。早期对话中的关键信息（用户名、偏好、决策）永久丢失，而 50 条消息对于 10 轮多步骤对话来说转瞬即逝。

**根因**：裁剪逻辑是纯数据结构操作，没有利用 LLM 的理解能力做语义压缩。

**解决方案**（3 文件）：
- 新建 `ConversationSummarizer`：用 LLM 将中间消息压缩为 ≤200 字摘要，作为 `SystemMessage` 插入到首条之后、最近 N 条之前
- `ToolCallAgent` 新增 `summarizer` 字段，`think()` 中裁剪逻辑改为 `summarizer.compress(chatClient, messageList, 20)`
- `AgentSessionManager` 构造注入 + `wireAgentPersistence()` 中注入给 Agent
- 降级：LLM 摘要失败时回退到简单截断

**为什么不是向量存储**：实现摘要的内存/开发成本极低（~60 行代码），效果立竿见影；向量存储需要引入 PGVector + 额外的检索步骤，改动面大。摘要作为"轻量级长期记忆"是当前阶段的最优解。

### #36 显式规划阶段（对标 LangGraph Plan-Execute / Claude Code TODO）

**问题**：Agent 收到复杂任务（如"研究 AI Agent 进展，搜索文章，总结，生成 PDF"）后直接进入 Think→Act 循环。没有"先想清楚做什么"的阶段，容易在中间步骤迷失方向——忘记原始目标、做无关搜索、不知下一步该干什么。

**根因**：ReAct 模式的核心缺陷——每步只看到上一步的结果（马尔可夫链），缺乏全局视野。

**解决方案**（4 文件）：
- 新建 `Plan` record（`PlanStep` 含 goal / toolHint / successCriteria）
- 新建 `SimplePlanner`：用一次轻量级 LLM 调用生成 JSON 格式子步骤计划
- `DevAssistantAgent` 新增 `setPlanIntoPrompt()`：计划文本追加到 system prompt
- `AiController`：新会话时（messageList ≤ 1）调用 planner，规划失败静默降级
- 简单任务（< 20 字）跳过规划

**设计决策**：使用 Prompt Engineering 实现轻量级规划（注入到 system prompt），而非重构为 Plan-Execute 分离架构。优势是改动量极小（~100 行），不改变 Agent 循环核心逻辑。

### 本轮经验教训

1. **内部流式 + 外部同步是过渡方案的最佳实践**：#34 用 `blockLast()` 保持 `step()` 同步语义，避免了全链路重构。这适用于任何"外层是 while 循环，内层想改成流式"的场景。

2. **Prompt Engineering 是成本最低的架构改进**：#36 不需要新 Agent 类型、新 Controller、新数据库。只是在 system prompt 前面加了 200 个字的计划。但效果——Agent 执行聚焦度——可能远超实现成本。

3. **concurrent 数据结构要提前设计**：#33 的 `HashMap` → `ConcurrentHashMap` 改动只有一行，但如果上线后才暴露线程安全问题，排查会非常困难。任何会被多线程访问的共享状态，从一开始就用并发安全的数据结构。

4. **LLM 摘要胜过简单截断**：#35 的核心洞察。50 条消息截断后丢掉的可能是"用户叫小羊"这种关键信息。LLM 摘要虽然多花一次 API 调用，但保留了语义——投入 1 次 API 调用，保住 30 轮对话的知识。

## 七、第六轮诊断修复 — 端到端测试 + 前端事件链路

> 日期：2026-07-04 | 发现并修复 6 个问题（#37-#42）

### #37 GlobalExceptionHandler 缺失 Spring MVC 异常处理

**问题**：`POST /cancel` 缺少 `sessionId` 参数返回 500；`PATCH /sessions/{id}` 缺少 body 返回 500；`POST /rag/documents`（GET-only 端点）返回 500。全部应返回 4xx。

**根因**：`GlobalExceptionHandler` 作为新文件，只处理了业务层的 `IllegalArgumentException` 和 `IllegalStateException`，再接一个泛化 `Exception`→500 兜底。Spring MVC 框架在请求解析阶段（参数绑定、Body 读取、Method 匹配）抛出 `MissingServletRequestParameterException` / `HttpMessageNotReadableException` / `HttpRequestMethodNotSupportedException` 等，全部落入泛化 handler → 500。

```
DispatcherServlet 请求解析失败 → Spring MVC 内置异常
  → @RestControllerAdvice 遍历 handler
  → IAE handler? 不匹配 → ISE handler? 不匹配
  → Exception handler? 匹配 → 500  ← bug
```

**解决方案**（`GlobalExceptionHandler.java`，+80 行）：新增 4 个 handler 夹在业务异常与泛化 Exception 之间：
| 异常 | HTTP 状态 | 触发场景 |
|------|----------|----------|
| `MissingServletRequestParameterException` | 400 | 缺必填 `@RequestParam` |
| `HttpMessageNotReadableException` | 400 | 缺/错 `@RequestBody` |
| `MethodArgumentTypeMismatchException` | 400 | 参数类型转换失败 |
| `HttpRequestMethodNotSupportedException` | 405 | 对 GET-only 用 POST |

**验证**：前测全 500，修复后精确返回 400/405 + 中文错误描述。

**教训**：`Exception` catch-all 是双刃剑——兜底未知错误的同时会吞掉框架层的客户端错误。正确的三层结构：框架异常 → 4xx、业务异常 → 4xx/409、未知异常 → 500。

### #38 `streamingMsgIndex` 暂时性死区导致 Agent 无响应

**问题**：用户发送 "你好"，聊天面板出现用户消息后思考链永远转圈。Agent 无任何输出，EventSource 从未建立连接。

**根因**：`AgentChat.vue:182` `streamingMsgIndex = -1` 引用 `let streamingMsgIndex` 位于同函数 189 行。`let` 有 TDZ——声明前访问抛出 `ReferenceError`。

```javascript
// sendMessage() 函数内
connectionStatus.value = 'connecting'  // ✅ L179 执行成功
agentEvents.value = []                 // ✅ L180 执行成功
streamingMsgIndex = -1                 // 💥 L182 ReferenceError
// 以下代码永不执行
const url = ...                        // ❌ 未执行
eventSource = new EventSource(url)     // ❌ 未创建
```

用户看到的效果：消息已发送、转圈已开始，但因函数在建立连接前崩溃，连接从未建立。`onerror` 也不会触发（无 EventSource 对象）。

**为什么 `API_BASE`（L183 引用 L267 的 `const`）不报错？** 因为 `API_BASE` 是模块顶层 `const`，模块已加载完成，TDZ 已结束。而 `streamingMsgIndex` 是函数内 `let`，在函数作用域内处于 TDZ。

**解决方案**（`AgentChat.vue`）：声明提至模块顶层（L124），与 `eventSource` 同级。

**教训**：`let`/`const` 的 TDZ 不仅是学术概念——一个 `=` 赋值与 `let` 声明之间的 7 行距离就能让 EventSource 静默失效。ESLint 规则 `no-use-before-define` 可防止此类问题。

### #39 `thinking` 占位文本泄漏到聊天面板

**问题**：聊天面板中 AI 回复以 "正在调用 LLM 分析当前状态..." 开头，后端内部状态描述暴露给用户。

**根因**：`AgentChat.vue` 对 `thinking` 事件的处理是 `finalAnswerText = evt.content`（"正在调用 LLM 分析当前状态..."）。然后 `thinking_delta` 追加实际内容（"你好，有什么可以帮你？"），最终 `finalAnswerText` = "正在调用 LLM 分析当前状态...你好，有什么可以帮你？"——占位符混入用户可见消息。

后端 `thinking` 事件语义：它是对前端的进度通知（"LLM 开始思考了，准备接收 deltas"），不携带任何需要展示给用户的内容。

**解决方案**（`AgentChat.vue`）：删除 `thinking` 事件中 `finalAnswerText` 的赋值逻辑，仅 `thinking_delta` 和 `final_answer` 参与聊天消息构建。

### #40 跨步骤 `finalAnswerText` 累积

**问题**：多步骤任务中，Step 2 的流式内容追加在 Step 1 已累积的内容之后，导致回复中出现重复。

**根因**：`finalAnswerText` 在函数开头初始化为 `''`，但在 `onmessage` 闭包中从未在新步骤开始（`step_start`）时重置。ReAct 模式中多个 Think→Act 循环会产生多组 `thinking_delta`，全部追加到同一变量。

**解决方案**（`AgentChat.vue`）：新增 `step_start` 事件处理：`finalAnswerText = ''; streamingMsgIndex = -1`。

### #41 `thinking` 事件覆盖思考链内容

**问题**：思考链中 Step 2 的真实思考（deltas 构建）被第二个 `thinking` 事件（"已连续 2 步未调用工具，自动结束"）直接替换。

**根因**：`ThinkingChain.vue` `case 'thinking'` 无条件赋值 `s.thinking = event.content`，未区分两种场景：

| 场景 | 事件顺序 | thinking 语义 |
|------|----------|---------------|
| A | `thinking` → `delta × N` | 占位符（"正在调用 LLM..."） |
| B | `thinking` → `delta × N` → `thinking` | 占位符 → 真实内容 → 总结 |

场景 B 中第二个 `thinking` 覆盖了 deltas 构建的真实内容。

**解决方案**（`ThinkingChain.vue`）：
- `thinking_delta`：首条 delta 检测到占位文本 `'正在调用 LLM 分析当前状态...'` 时先清空
- `thinking`：若 `s.thinking` 已有内容 → 视为总结，写入 `s.summary` 而非覆盖

### #42 `step_end` 空 content 覆盖 summary

**问题**：`step_end` 的 `content` 为空字符串时，上方 `thinking`（总结）写入的 `s.summary` 被清空。

**根因**：`step_end` handler 无条件 `s.summary = event.content`，当内容为空时就等于 `s.summary = ''`。

**解决方案**（`ThinkingChain.vue`）：仅当 `event.content` 非空时才赋值。

### 本轮经验教训

1. **let 声明应放在函数顶部**：#38 是一个"改动只有一行，影响整个功能"的 bug。ESLint 的 `no-use-before-define` 规则可以静态检测。Vue `<script setup>` 中模块级变量应集中声明在顶部。

2. **区分"进度通知"和"内容事件"**：#39、#41 的核心教训。后端 `thinking` 事件携带的是进度信息（"正在调用 LLM..."），不应作为显示内容。前端收到事件时必须理解其语义，不是所有 `event.content` 都应该展示。

3. **EventSource 的生命周期需要防御编程**：#38 中 EventSource 创建失败后 `onerror` 无法触发（无对象可绑）。应该在创建前就先设置好错误处理策略，或用 try-catch 包裹整个 sendMessage 函数。

### #46 🔴 `stream()` 的 `blockLast()` 返回空 chunk 导致消息重复（本轮最严重）

**发现过程**：用户测试"你好"后发现回复出现两遍——这是从真实对话中直接观察到的症状。

**根因**：Token 级流式改造（#31）使用 `Flux<ChatResponse>` + `blockLast()`：
```
chunk 1: getText() = "你好"
chunk 2: getText() = "！有什么我可以帮你的吗？"
chunk N: getText() = ""   ← blockLast() 返回空结束标记
```

`assistantMessage.getText()` 拿到的是最后一个 chunk 的文本（空），Agent 认为 LLM 没输出 → 不设 FINISHED → 继续 Step 2 → LLM 又说一遍。

**调用链路**：
```
think()
  ├─ stream().chatResponse() → Flux<ChatResponse>
  ├─ doOnNext(chunk) → fullThinking.append(delta) // ✓ 文本已累积
  ├─ blockLast() → chatResponse                    // 最后一个 chunk
  ├─ assistantMessage.getText() → ""              // ✗ 空！这是结束标记
  ├─ thinkingText 为 blank
  ├─ Agent 不识别为最终答案 → 不设 FINISHED
  ├─ step() 返回 "思考完成 - 无需行动"
  ├─ Step 2: 同样的流程 → LLM 又说一遍 → 重复回复！
  └─ getFinalAnswerText() 在 messageList 中找不到有文本的 AssistantMessage
     → final_answer 不发射 → 聊天面板无最终回复文本
```

**解决方案**：
1. 用 `fullThinking.toString()` 替代 `assistantMessage.getText()` 作为 `thinkingText` 主来源
2. 删除重复的 `thinkingText` 提取逻辑，统一在 `blockLast()` 后计算
3. `messageList.add()` 时若 `assistantMessage.getText()` 为空，创建 `new AssistantMessage(thinkingText)` 替代
4. `getFinalAnswerText()` 现在能在 messageList 中找到完整文本 → `final_answer` 正常发射

**涉及文件**：`ToolCallAgent.java`

**验证结果**：重启后端后测试"你好"，事件流从 2 步缩减为 1 步，`final_answer` 正常发射，前端显示单条回复。

### 本轮经验教训

1. **多 Provider 测试矩阵不可或缺**：#43 在 OpenAI/Anthropic 上不存在，只在 DashScope 上暴露。功能添加后必须在所有 Provider 上验证。

2. **错误报告用系统通道，不用对话通道**：#44 的教训。`[SYSTEM NOTE: ...]` 比 `AssistantMessage("出错了")` 正确得多——LLM 能理解前者是系统注入的上下文提示。

3. **回退文案不能撒谎**：#45。"任务完成"是假设，不是事实。默认行为应该是"我不确定发生了什么，请检查"。

### 诊断方法

在前七轮修复基础上，对 `think()` → `act()` → 各工具实现进行全链路代码路径追踪，识别所有可能的工具调用失败点。然后用 `qwen-plus-2025-07-28` 模型运行 10 条真实用户对话场景测试。

### 发现的问题

#### #47（严重 — P0）：失败的工具调用计入 `sameToolCallCount`，导致工具被永久封禁

**根因**：`think()` 中用 `merge()` 在工具执行前递增计数，但工具执行在 `act()` 中。若执行失败（网络超时、Bing 屏蔽等），计数已 +1。5 次失败后工具被永久封禁，LLM 无工具可用，任务失败。

**致命循环**：
```
Step 1: think() → merge → count=1 → act() → Bing超时 → 失败
Step 2: think() → merge → count=2 → 又失败
...
Step 6: think() → merge → count=6 > 5 → 🚫 过滤 → LLM搜索能力被剥夺
```

**修复**（`ToolCallAgent.java`）：
1. `think()` L210：`merge()` → `getOrDefault()`（只读不写）
2. 计数移到 `act()` — 工具实际执行后才递增
3. 新增 `consecutiveToolFailures` 独立计数器 — 连续失败 ≥ 3 次通过 `retryHint` 通知 LLM 换方案

```java
// think() — 只读
int count = sameToolCallCount.getOrDefault(tc.name(), 0); // 不再 merge

// executeSingleToolCall() — 成功时计数
sameToolCallCount.merge(toolCall.name(), 1, Integer::sum);
consecutiveToolFailures.remove(toolCall.name()); // 成功清零

// executeSingleToolCall() — 失败时独立计数
int fails = consecutiveToolFailures.merge(toolName, 1, Integer::sum);
if (fails >= 3) {
    retryHint = "工具已连续失败3次，请停止使用此工具...";
}
```

#### #48（P1）：单工具路径执行失败被静默吞掉

**根因**：`toolCallingManager.executeToolCalls()` 内部异常时，`extractToolResponseMessage()` 返回 null → `allResponses` 为空 → 返回"工具执行完成，但未获取到返回结果"，无实际错误原因。

**修复**：null/空时构造明确的错误 `ToolResponseMessage`，含工具名和"请勿重试"提示。

```java
if (trm == null || CollUtil.isEmpty(trm.getResponses())) {

```

#### #49（P2）：多工具并行超时/异常被静默丢弃

**根因**：`CompletableFuture.get(30s)` 超时 → null → `.filter(r -> r != null)` 丢弃。

**修复**：超时/异常时构造错误 ToolResponse（含 tc.id/name），不再返回 null。

```java
} catch (TimeoutException e) {
```

#### #50（P1）：工具错误被伪装成成功响应

## 十、第九轮：搜索精准度提升 + Tavily API 接入（2026-07-04）

### 问题诊断

Agent 工具搜索结果精准度不高，根因涉及三个环节：

| 环节 | 现状 | 精准度损失 |
|------|------|-----------|
| WebSearch | 单一 Bing HTML 抓取 | cn.bing.com 反爬检测，返回通用缓存而非真实搜索结果 |
| WebScraping | `document.body().text()` 全量提取 | 导航/广告/侧栏噪音 > 50%，8KB 截断后有效正文不足 |
| 后处理 | 无去重/质量评分/排序 | 原始结果直接丢给 LLM，LLM 在噪音中找信息 |

### 已实施改进

#### 1. ContentExtractor 正文提取（方案 B）

新增 `ContentExtractor.java` — 基于 Jsoup 的简化版 Readability 算法：

```
HTML → 去除 <script>/<style>/<nav>/<footer>/<aside> 等噪音标签
     → 定位主内容区（<article> → <main> → 文本密度打分）
     → 提取纯文本（保留段落结构）
     → 截断至 8000 字符
```

文本密度公式：`score = ownTextLength / (1 + linkTextLength × 2)`

## 十二轮：跨会话记忆分析（2026-07-04）

### 问题现象

用户反馈："跟 Agent 说'我是谁'，Agent 回答'我不知道你是谁'"。

经排查：用户有 2 个活跃会话，每个会话仅 2 条消息（"我是谁" → "我不知道"）。用户从未在任何会话中自我介绍过。

但这暴露了一个架构层面的问题：**会话级记忆的天然局限**——即使用户在会话 A 中说过"我叫小明"，切换到会话 B 后 Agent 依然不认识他。

### 根因分析

当前系统采用**会话级记忆**（session-scoped memory）：
- 每个 `sessionId` 拥有独立的 `messageList`，持久化到独立的 `.json` 文件
- `getOrCreateSession()` 通过 sessionId 恢复对应会话的历史
- 会话之间完全隔离，无共享记忆

会话隔离的触发场景：

| 触发条件 | 后果 |
|---------|------|
| 清浏览器缓存 | localStorage 中 sessionId 丢失 → 新会话 |
| 点击"新对话"按钮 | 强制生成新 sessionId |
| 换设备/浏览器 | 没有 sessionId → 新会话 |
| Settings 页配置 API Key | 可能触发 sessionId 更新 |

### 主流产品对比

| 产品 | 记忆方案 | 机制 |
|------|---------|------|
| **ChatGPT** | 用户级记忆 | GPT-4o 自动提取关键信息存入 Memory，跨会话生效，用户可在设置中管理 |
| **Claude** | 项目级指令 | 用户手动在 Project Settings 中写 instructions，不自动跨会话记忆 |
| **Gemini** | 会话级 | 每会话独立，无自动跨会话 |
| **开源 Agent** | 会话级 + 可选 | 开发者集成向量数据库做长期记忆 |

核心分歧：ChatGPT（隐式自动）vs Claude（显式手动）vs 开源（自己搭）。

### 可选方案对比

| 方案 | 复杂度 | 隐私风险 | 用户体验 | 推荐度 |
|------|--------|---------|---------|--------|
| A：保持会话级 | 零 | 零 | 差（跨会话失忆） | ★★★ 当前选择 |
| B：轻量用户画像 | 低（~50行） | 低 | 中（手动控制） | ★★ 未来可做 |
| C：向量长期记忆 | 高 | 高 | 好（自动） | ★ 学习项目太重 |

**决策**：保持方案 A（会话级记忆），理由：
1. 当前会话内记忆已验证正常工作——同 sessionId 下 Agent 完全能记住用户
2. 跨会话画像引入隐私风险（用户信息发送给 LLM 提供商）
3. 主流产品也在此问题上分歧，没有"标准答案"
4. 更好的学习路径：先打磨会话内记忆，再考虑扩展

**未来如需实现**：推荐方案 B（轻量画像），对齐 Claude 的显式控制路线——用户明确知道 Agent 记住了什么，用 `/whoami` 查看、`/forget` 清除。

### 关键教训

1. **用户不一定知道"会话"的概念**：非技术用户期望 Agent "记住我"，不理解 sessionId 的隔离机制。
2. **ChatGPT 设置了错误的期望**：用户习惯了 ChatGPT 的"记住一切"，但自动记忆的隐私代价往往被忽略。
3. **显式控制 > 隐式自动**：对个人项目，让用户明确知道 Agent 记住了什么、能随时清除，比自动提取更安全。

## 十三轮：记忆会话管理优化 — 编码修复 + 资源泄漏 + 真机验证

### 问题发现

通过真机模拟 4 轮中文对话压力测试 + 全代码审查，发现 8 个优化点：

| # | 优先级 | 问题 | 根本原因 |
|---|--------|------|---------|
| 1 | **P0** | 中文 URL 参数编码损坏 | `server.tomcat.uri-encoding` 未显式声明 UTF-8，Windows 环境默认非 UTF-8 导致 38% 磁盘文件中文乱码 |
| 2 | **P1** | `cancelFlags` Map 只增不减 | `clearCancelFlag()` 定义了但从未被调用，Agent 结束后 key 不清理 |
| 3 | **P1** | 无内存会话上限 | 磁盘配额 20 个但内存无限制，100 用户同时创建 → 100 个 Agent 都在内存 |
| 4 | **P2** | 冗余 ArrayList 拷贝 | `save()` 中 `new ArrayList<>(messages)` 防御性拷贝后 `writeJsonArray` 内部又构建新 List |
| 5 | **P2** | `lastPersistedSize` 未同步 | `loadSessionHistory()` 恢复消息后未更新计数器，导致恢复后首次写入冗余 |
| 6-8 | **P3** | 设计瑕疵 | 双文件无事务、`isActive` 语义模糊、`listSessions()` 有副作用 |

### 修复方案

#### 1. UTF-8 编码（`application.yml`）

```yaml
server:
  servlet:
    encoding:
      charset: UTF-8
      enabled: true
      force: true
  tomcat:
    uri-encoding: UTF-8
```

**关键认知**：URL query parameter 编码 ≠ request body 编码。Spring 的 `CharacterEncodingFilter` 只控制 body，query string 由 Tomcat Connector 独立控制。

#### 2. cancelFlags 生命周期（`BaseAgent.java` + `AgentSessionManager.java`）

新增 `onCleanupCallback`（参照已有 `onPersistCallback` 模式），Agent 生命周期结束时自动清理：
```
Agent.runStreamStructured()
  → step loop
  → finally { cleanup() }
    → onCleanupCallback.run()
      → cancelFlags.remove(sessionId)
```

#### 3. lastPersistedSize 同步（`BaseAgent.java`）

```java
public void loadSessionHistory() {
    // ... restore from disk ...
    this.messageList = saved;
    this.lastPersistedSize = saved.size();  // ← 新增
}
```

### 真机验证结果

**18/18 全部通过**：

- ✅ 4 轮中文对话 + 工具调用
- ✅ Agent 跨轮次记忆：张伟/28岁/杭州/后端架构师/篮球/摄影
- ✅ 搜索工具：3 次 web_search + 结果降级
- ✅ 上下文交叉引用：结合搜索结果推荐 AI 学习方向
- ✅ 持久化：14 条消息完整写入磁盘，中文无乱码
- ✅ Meta 数据：messageCount=14，标题正确，时间戳准确

### 关键教训

1. **分开控制 URL 和 Body 编码**：`server.tomcat.uri-encoding` ≠ `server.servlet.encoding.charset`
2. **回调模式管理外部资源**：`onPersistCallback` + `onCleanupCallback` 让 Agent 自治
3. **资源要有上限**：`Map` 不加 maxSize = 隐藏的内存泄漏
4. **真机测试发现代码审查遗漏的问题**：同样的代码 inline 执行 vs 文件执行可能有不同行为（URL 中的 `/ai` 漏掉导致全 500）

## 十四、会话恢复体验优化（2026-07-04）

### 问题

用户反馈：刷新页面后聊天面板出现"— 已恢复 16 条历史消息 (磁盘) —"，体验粗糙。

### 根因分析

这条消息**不是 LLM 生成的**，是前端 `AgentChat.vue:152` 硬编码的：

```javascript
// loadSessionHistory() 中
if (ctx.messageCount > 0) {
  addMessage(`— 已恢复 ${ctx.messageCount} 条历史消息 (${source}) —`, false, 'ai-system')
}
```

`isUser: false` 导致消息被渲染在 AI 气泡中，看起来就像 Agent 的回复。同时 header 中还有 `<span class="source-tag">磁盘恢复</span>` 标签。

### 为什么这是问题

1. **泄露实现细节** — 用户不需要知道消息是从"磁盘"还是"内存"恢复的
2. **打破对话沉浸感** — 系统消息打断正常对话流
3. **与主流产品不一致** — ChatGPT/Claude/Gemini 都没有这种行为

### 主流产品对比

| 产品 | 做法 |
|------|------|
| **ChatGPT** | REST API 拉取消息 → 立即渲染 → 无"恢复"提示 → 对话自然呈现 |
| **Claude** | 同上，侧边栏"Resume"按钮 → 静默加载全部历史 |
| **Gemini** | 同上，按日期分组 → 点击会话直接展示 |

**核心原则**："加载历史"是纯前端操作（REST API → DOM），"聊天"才涉及 LLM（SSE 流式）。两者不应混淆。

### 修复

**文件**：`ai-agent-frontend/src/views/AgentChat.vue`（4 处修改）

| # | 改动 | 类型 |
|---|------|------|
| 1 | 删除 `addMessage("已恢复 X 条历史消息...")` | 移除实现细节泄露 |
| 2 | 删除 header `<span>磁盘恢复</span>` 标签 | 同上 |
| 3 | 删除 `sessionSource` ref 变量及全部赋值 | 清理死代码 |
| 4 | 删除 `.source-tag` CSS 规则 | 清理死样式 |

### 改后效果

```
│ 用户: 帮我搜索...            │        │ 用户: 帮我搜索...            │
```

### 关键教训

1. **不要在前端插入"系统消息"来宣布架构行为** — 每一条用户可见的消息都应该是对话的自然组成部分

### 消息溢出修复（2026-07-05）

**现象**：Agent 长回复中的 emoji（✅🌟）、URL、Markdown 列表等符号超出会话气泡左边栏。

**诊断**：
1. flex 子元素默认 `min-width: auto`（=内容最小宽度），导致 `.msg-bubble` 不遵守父容器 `.msg` 的 `max-width: 85%`
2. `word-break: break-word` 已过时且处理 emoji/URL 断行不彻底
3. `.markdown-body pre code` 未设置 `white-space: pre`，代码块格式可能被破坏

**修复**（ChatRoom.vue）：
- `.msg-bubble`：`min-width: 60px` → `min-width: 0` + 新增 `overflow: hidden`（核心修复）
- `.msg-text`：新增 `overflow-wrap: anywhere`（替代过时的 `word-break: break-word`）
- `.markdown-body pre`：新增 `max-width: 100%`
- `.markdown-body pre code`：新增 `white-space: pre; overflow-wrap: normal`（代码保持格式 + 横向滚动）
- `.markdown-body`：新增 `ul/ol/li` 样式规范和缩进

**微调**（AgentChat.vue）：
- `.sidebar` 宽度 `260px` → `300px`，会话标题有更多展示空间

**经验**：flex 布局中 `overflow: hidden` 要生效，必须同时设置 `min-width: 0`（或 `overflow` 为非 `visible` 的值也可以隐式设置，但显式更清晰）。

## 十六、第十七轮：提示词架构调整 — 全权 LLM 自主决策（2026-07-05）

### 背景

用户提出核心架构原则："不编排任何使用流程，全权交由 LLM 处理，当 LLM 分析用户需求时，觉得需要再去调用工具即可。"

本轮将 Agent 从"编排式提示词"架构迁移为"全权 LLM 自主决策"架构。

### #51 模型缺乏元认知自我评估能力

**问题**：使用 `qwen-plus-2025-07-28`，即使精简 Prompt 也无法让 LLM 自主判断"信息够了没"。每步都倾向于继续搜索，因为模型不理解"充分性"概念。

**根因**：`qwen-plus-2025-07-28` 是较早的快照版本，不具备元认知（meta-cognition）能力——即"反思自己的输出是否已经满足用户需求"。

**解决方案**（`application.yml:16`）：
```yaml
# 改前
model: qwen-plus-2025-07-28
# 改后
model: qwen-plus-latest
```

**效果**：模型能自行判断搜索信息的充分性，自然减少不必要的搜索步骤。

### #52 工具结果截断过短导致 LLM 信息不足

**问题**：MAX_TOOL_RESULT_LENGTH=800（第 21 轮设定的值），Tavily 返回的正文摘要（含具体数据、JEP 编号、来源）被大量截断。LLM 看不到完整信息 → 无法判断"够了没" → 重复搜索尝试获取更多信息。

**根因**：第 21 轮优化追求"缩短思考链"，将上限从 2000 降为 800。但对搜索工具而言，800 字符只能容纳 1-2 条结果的片段，LLM 在做"盲人摸象"式的决策。

**解决方案**（`ToolCallAgent.java:54-58`）：
```java
// 改前
private static final int MAX_TOOL_RESULT_LENGTH = 800;
// 改后
private static final int MAX_TOOL_RESULT_LENGTH = 4000;
```

**关键认知**：工具结果截断和 LLM 决策质量是一对矛盾。截断太短 → LLM 信息不足 → 多步重试 → 总思考链更长。适当放宽反而减少总步骤。

### #53 System Prompt 编排式指令抑制 LLM 自主性

**问题**：System Prompt（~34 行）包含了知识策略、搜索条件、停止规则等编排指令：
```
== Knowledge Strategy ==
DEFAULT: Use your own knowledge to answer directly.
SEARCH ONLY WHEN: real-time news / specific facts / verification
STOP SEARCHING WHEN: 3 searches poor → use own knowledge
```

虽然这些指令是为了防止搜索死循环，但它们本质上是在"替 LLM 做决策"。

**根因**：第 8/10 轮优化时，为解决搜索死循环和思考链过长问题，在 Prompt 中加入了大量行为指导。这些指令有效但违背了"全权自主"的设计哲学。

**解决方案**（`DevAssistantAgent.java:87-103`）：

```java
// 改造后：零编排指令
String SYSTEM_PROMPT = """
        你是一个 AI 智能助手。

        当前时间: %s

        你可以使用工具来搜索网页、抓取内容、读写文件、执行终端命令、
        生成 PDF 和下载资源。根据用户的需求自行判断是否需要调用工具。

        始终使用中文回复。
        """;
this.setSystemPrompt(SYSTEM_PROMPT);
this.setNextStepPrompt(null);  // ★ 移除每步自检注入
```

**关键变化**：
- ✅ 移除知识策略（DEFAULT/SEARCH ONLY WHEN/STOP SEARCHING WHEN）
- ✅ 移除 `NEXT_STEP_PROMPT`——不再每步注入"determine the next action"
- ✅ 只告知工具的存在，不指导何时用、何时停
- ✅ `ToolCallAgent.think()` 检测到 `nextStepPrompt == null` 时完全不注入 per-step instruction

### #54 thinking_delta 泄漏到聊天面板

**问题**：前端 `AgentChat.vue` 中 `thinking_delta` 事件同时创建流式聊天消息，LLM 的推理过程（"我需要先搜索 AI 相关新闻，看看有哪些最新动态..."）被用户看到。

**根因**：第 5 轮 Token 级流式改造时，`thinking_delta` 的处理逻辑是"既追加到思考链，又在聊天面板创建流式消息"。当时认为流式 token 就是最终回复的一部分。

但 `thinking_delta` 中可能包含 LLM 的内部推理（尤其是启用 extended thinking 或 reasoning 的模型），这些内容不应该出现在用户聊天中。

**解决方案**（`AgentChat.vue:195-232`）：

- 删除 `thinking_delta` → 聊天消息的创建路径
- 聊天面板只展示 `final_answer` 事件（LLM 的最终回复）
- 思考链面板通过 `agentEvents` 获取所有事件（包括 `thinking_delta`）
- 同时清理死代码：`streamingMsgIndex`、`finalResponse` 变量

**设计原则**：
- **思考链 = 内部过程**：thinking/thinking_delta/tool_call/tool_result
- **聊天面板 = 对外回复**：final_answer 仅此一个来源
- 两者通过不同的数据通道消费相同 SSE 事件流，前端做过滤

### 架构反思

```
人工编排流程     ←—————— 光谱 ——————→     全权 LLM 自主
高效、可预测                        灵活、但行为不可控
用户知道会得到什么                   用户不知道 LLM 会做什么
适合生产环境                        适合探索/实验
```

当前版本选择了全自主端。这意味着：
- LLM 可能做用户没预期的事（如主动下载 PDF）
- 步骤数不完全可控
- 行为随模型版本变化

**核心价值**：这种架构下，Agent 的行为是其"智能"的真实体现，而非被 Prompt 编排的"木偶"。对学习 AI Agent 来说，这是更本质的观察窗口。

### 关键经验

1. **工具质量 > Prompt 编排**：提升工具结果质量（Tavily API + 4000 字符上限）比精心设计"3 次无果即停"更有效——LLM 看到好结果后自然会停。

2. **模型的元认知能力决定自主性的上限**：`qwen-plus-2025-07-28` 换了精简 Prompt 也无法自主判断"够了没"，换模型解决了所有 Prompt 解决不了的问题。

3. **思考链和聊天面板必须严格分离**：两者虽然共享同一个 SSE 数据源，但语义完全不同。前端做过滤，后端不同的事件类型本身就对应不同的展示目标。

4. **自主 Agent 是模型能力的真实镜像**：移除 Prompt 编排后，Agent 的行为完全反映模型的真实能力——包括它的判断力、知识边界、以及它认为"够好"的标准。
